# Billions English 项目设计与演进回顾

## 文档目的

这份文档用于系统回顾 `Billions English` 从原型到当前版本的设计思路、架构演进、关键问题、修复方案与后续改进方向，方便后续继续迭代时快速建立上下文，避免重复踩坑。

---

## 一、项目背景与目标

### 1.1 产品目标

项目最初的目标不是做一个泛化英语学习工具，而是做一个围绕《亿万》（Billions）台词素材的 **高语境英语学习应用**。核心思路是：

- **真实语境**：用影视台词替代脱离语境的单词表
- **学习闭环**：学习 → 复习 → 测验 → 苏格拉底追问 → 统计反馈
- **高阶表达训练**：不仅记词义，还训练语气、句式、迁移表达和商务/法律/金融语感
- **长期可坚持**：通过每日任务、XP、连续学习、SRS 复习建立习惯

### 1.2 目标用户

- 想通过美剧提升英语实际表达能力的学习者
- 对商务英语、谈判表达、金融/法律语境感兴趣的中高级用户
- 希望在网页或手机上持续学习的人群

### 1.3 产品约束

项目在早期就是一个 **单仓库原型产品**，因此设计上有几个显著约束：

- 先求可用，再求完备
- 优先本地可运行，不依赖复杂后端
- 优先 Web 原型验证，再补移动端
- AI 能力必须能被真实使用，而不是停留在“预留接口”

## 二、当前系统设计总览

### 2.1 技术架构

当前系统采用以下技术方案：

- **前端**：React 19 + Vite
- **路由**：`react-router-dom` + `HashRouter`
- **样式**：原生 CSS 主题体系
- **持久化**：浏览器 `localStorage`
- **AI 连接层**：本地 Node HTTP Bridge + 外部 CLI 模型
- **移动端封装**：Capacitor + iOS 工程

### 2.2 目录与职责划分

#### 页面层

- **`App.jsx`**：应用 Shell、主路由分发
- **`Dashboard.jsx`**：学习首页 / 每日任务 / 学习总览
- **`LearnPage.jsx`**：逐句学习台词、集数导航、进度追踪
- **`ReviewPage.jsx`**：SRS 间隔复习
- **`QuizPage.jsx`**：考试/测验
- **`SocraticPage.jsx`**：AI 苏格拉底式追问学习
- **`VocabularyPage.jsx`**：词汇本
- **`StatsPage.jsx`**：统计与学习反馈
- **`SettingsPage.jsx`**：学习偏好与 AI Bridge 配置

#### 数据层

- **`src/data/episodes.js`**：核心剧集与场景数据入口
- **`src/data/seasonOneExpansion.js`**：第一季扩展内容

每个场景（scene）基本包含：

- `character` / `characterCn`
- `context` / `contextCn`
- `dialogue` / `translation`
- `keywords`
- `grammar`
- `difficulty`
- `tags`

这套结构是整个产品的数据中心：学习、测验、复习、苏格拉底追问都建立在 scene 上。

#### 学习引擎层

- **`src/utils/storage.js`**：学习进度、SRS、测验记录、每日任务、设置、连续学习、苏格拉底历史
- **`src/utils/quizGenerator.js`**：组卷与评分
- **`src/utils/socratic.js`**：苏格拉底阶段、模板、质量评估、阶段推进策略
- **`src/utils/ai.js`**：Bridge 通信与 Socratic 系统提示词构造

#### AI Bridge 层

- **`server/ai-bridge.js`**：本地 HTTP 服务，接收前端消息并转发给 CLI 模型

#### 移动端层

- **`capacitor.config.json`**：Capacitor 配置
- **`ios/`**：iOS 原生工程

### 2.3 当前核心学习闭环

当前产品已经形成比较完整的闭环：

1. **Learn**：学习新台词、理解句意、积累词汇和句式
2. **Review**：通过 SRS 做延迟复习
3. **Quiz**：用多题型测验检验提取能力
4. **Socratic**：通过追问式学习把“看懂”推进到“会分析、会迁移”
5. **Stats / Dashboard**：用 XP、连续天数、任务状态和测验结果反馈学习成果

### 2.4 关键设计决策

#### 使用 `HashRouter`

原因：

- 静态部署更简单
- 与 WebView / Capacitor 的兼容性更稳
- 避免服务端路由重写问题

#### 使用 `localStorage`

原因：

- 原型阶段零后端即可跑通完整学习链路
- 学习记录、SRS、设置、测验结果都可快速落地
- 降低首轮实现成本

代价：

- 无跨设备同步
- 不适合多端统一账号体系
- 不适合更复杂的分析与推荐逻辑

#### AI 不直接在前端持有模型能力

最终演化出的方案是 **前端只发起对话，真正执行模型调用的逻辑在本地 Bridge/CLI 层**。这样做有几个好处：

- 不要求用户一定有 API Key
- 不把模型敏感配置暴露到浏览器
- 可以复用用户已有的大模型 CLI（如 `gemini-internal`）
- 更适合 iOS WebView / 本地开发联调

### 阶段 2：AI 学习体系从“预设模板”走向“真实模型驱动”

随着产品体验推进，一个非常关键的问题暴露出来：**模板式苏格拉底对话不够真实**。它可以提问，但不够像真正的导师，也很难根据学习者回答动态追问。

#### 初始思路

最初系统更接近“前端持有 AI 配置”的模式，文档和变更记录里也保留了这一路径，例如：

- README 中提到 Gemini API Key 管理
- CHANGELOG 中记录 AI-Powered Socratic Learning

#### 演进原因

实际体验中，用户并没有现成的 Gemini Key，但拥有一个已经封装好的命令行模型工具：`gemini-internal`。

这推动系统从“浏览器直接依赖 API Key”演进到“本地 AI Bridge + CLI 模型”的方案。

#### 最终改进

- 新增 **`server/ai-bridge.js`** 本地桥接服务
- 默认命令改为 **`gemini-internal`**
- 默认使用 `-p` 传 prompt，`-o text` 获取纯文本输出
- 前端通过 **`src/utils/ai.js`** 统一调用 Bridge
- 设置页新增 Bridge 地址配置与健康检查
- `SocraticPage.jsx` 不再依赖浏览器内的 API Key，而是通过 Bridge 与本地模型通信

#### 这一阶段解决的问题

| 问题 | 原状 | 改进 |
|---|---|---|
| 用户没有 API Key | 真实 AI 功能不可用 | 支持本地 CLI 模型桥接 |
| 模型接入方式不灵活 | 强绑定浏览器配置 | 抽象成 Bridge 层 |
| 安全性一般 | 可能把模型信息放在前端 | 改为本地服务隔离 |
| iOS/WebView 联调复杂 | 前端直连模型不稳 | 统一走 HTTP Bridge |

### 阶段 3：苏格拉底学习主线收束与阶段化约束

用户在实际使用过程中提出了非常关键的产品反馈：

> 当前的学习会偏离英语学习主线，在苏格拉底追问中体现得特别明显。

这个反馈本质上指出了 AI 模块的第一个核心产品问题：

- AI 可能“会聊天”，但不一定“会教学”
- 如果没有明确约束，追问会偏离词汇、句意、句型和迁移这条英语学习主线

#### 针对这个问题的改进

在 `socratic.js` 与 `ai.js` 中做了系统性重构：

- 明确苏格拉底学习主线：**句意 → 词汇 → 句式 → 迁移应用**
- 在 session 中加入 guardrails，限制 AI 不偏离英语学习主题
- 为每个 phase 增加：
  - `goal`
  - `idealTurns`
  - `maxTurns`
  - `qualityThreshold`
- 新增 `decideSocraticPhaseAction()`，根据回答质量和轮次决定：
  - 继续追问
  - 转入下一个 phase
  - 完成会话
- 在 `getSocraticSystemPrompt()` 中加入 mode 机制：
  - `open`
  - `follow_up`
  - `transition`
  - `complete`

#### 这一阶段解决的问题

| 问题 | 根因 | 改进 |
|---|---|---|
| AI 容易偏题 | 缺少教学边界 | 加入英文学习 guardrails |
| 追问结构不稳定 | 缺少显式阶段控制 | 建立 comprehension / vocabulary / grammar 三阶段 |
| 没有自然收束 | 没有完成条件 | 增加质量阈值与轮次预算 |
| 聊天像通用助手 | prompt 太宽泛 | 按 mode 输出更像 tutor 的短反馈 + 单问题 |

### 阶段 4：苏格拉底会话结束逻辑修复

在上一阶段完成后，又暴露出一个更具体的体验问题：

> 学习关键点完成后，对话还会无限扩展，不能自然结束，也不能自然进入下一个部分。

#### 根因

- 虽然已有阶段概念，但结束条件还不够严格
- 会话推进和任务完成没有形成统一状态机

#### 改进

- phase 预算明确化：**理想 2 轮，最多 3 轮**
- 回答质量达到阈值后，允许更快结束当前阶段
- 当最后阶段完成时，生成 `complete` 模式回复，不再继续发问
- prompt 中显式加入规则：**不要无休止扩展一个 phase**

#### 结果

苏格拉底体验从“无限扩展的 AI 聊天”向“受控的高质量教学流程”收敛。

---

### 阶段 5：苏格拉底学习完成后每日任务未标记

用户继续使用后又发现：

> 当前的对话学习结束后，任务并没有被标记，不符合预期。

#### 根因

- `updateDailyTask('socratic')` 原本只在局部分支里调用
- 不同结束路径可能绕过这段逻辑
- 缺少 exactly-once 保护，既可能漏记，也可能重复记

#### 改进

在 `SocraticPage.jsx` 中统一收敛为一个状态驱动方案：

- 用 `sessionComplete` 作为唯一完成信号
- 用 `useEffect` 统一观察完成状态并落盘
- 用 `hasMarkedDailyTaskRef` 保证只记一次
- 会话历史通过 `addSocraticSession()` 更新后同步回到页面状态

#### 结果

苏格拉底会话的 **完成状态、历史记录、每日任务** 三者关系变得一致。

### 阶段 6：测验系统答题 bug 修复与鲁棒性补强

用户继续测试时又提出：

> 考试测验有 bug，我们无法答题。

这次问题非常重要，因为测验属于学习闭环中的核心环节，一旦答题链路异常，整个闭环就断了。

#### 根因 1：`generateWordMatching()` 原地修改 `scenes`

`quizGenerator.js` 中原本使用了：

- `scenes.sort(() => Math.random() - 0.5)`

这会 **直接修改传入数组本身**。由于同一个 `scenes` 数组会被多个题型生成器复用，这种写法会把共享数据打乱，导致后续生成器行为不可预期。

#### 根因 2：语法题可能无法生成足够选项

`generateGrammarChoice()` 早期只做了少量字符串替换：

- 有些句子替换后没有变化
- 结果会造成错误选项不足
- 最终出现 options 不完整甚至异常

#### 根因 3：组卷没有容错

`generateQuiz()` 里如果某一题生成异常：

- 整个测验生成链路可能被中断
- 前端拿到 `null` 或空题集后没有明确提示

#### 根因 4：词汇配对题交互不够健壮

旧版配对题：

- 只能不断往后填
- 用户选错后不方便撤销
- 已选含义没有清晰禁用态

#### 针对这轮问题的改进

- `generateWordMatching()` 改为 `[...scenes].sort(...)`
- 为 `keywords` 缺失场景加防御性处理
- 为 `generateGrammarChoice()` 增加更多变换策略
- 当干扰项不足时回退为词汇选择题
- `generateQuiz()` 单题生成加入 `try/catch`
- 某题失败时尝试用词汇题替补，而不是整个测验崩溃
- `startQuiz()` 增加空结果检查和错误提示
- 配对题支持取消已选配对
- 已使用的 meaning 显示禁用态，减少误操作

#### 结果

测验模块从“容易在边缘条件下失效”变成“单题异常不影响整体可答”的鲁棒实现。

### 阶段 8：测后分析、错题本与长期纠错闭环

在测验系统稳定后，新的产品问题很快暴露出来：

> 学生做完题以后，只知道自己对了几题、错了几题，但不知道为什么错，也没有机制保证这些错误会在未来被重新击中。

这说明旧版测验虽然能“测”，但还不能真正“教”。它缺少两层关键能力：

- **短期反馈层**：逐题分析、正确答案对比、解释、改进建议
- **长期纠偏层**：错题记录、记忆曲线调度、周期性复训

#### 针对这个问题的改进

- 在 `QuizPage.jsx` 中重写结果页：
  - 增加错题分析 / 全部题目两栏标签页
  - 增加可展开的逐题详情卡片
  - 显示原文语境、用户答案 vs 正确答案、解析、改进建议
  - 增加按题型聚合的薄弱点分析
- 在 `storage.js` 中新增独立错题本：
  - 使用 `billions_english_wrong_answers` 持久化错题
  - 记录错题次数、首次/最近错误时间、SRS 调度字段、掌握状态
- 在 `quizGenerator.js` 中新增 `generateWrongAnswerQuiz()`：
  - 从到期错题生成复习 quiz
  - 优先复用原始错题所属场景
- 在测验提交时接入错题本与 SRS：
  - 新错题自动入库
  - 错题复习 quiz 会反向更新该错题的掌握进度

#### 这一阶段解决的问题

| 问题 | 原状 | 改进 |
|---|---|---|
| 测后反馈过浅 | 只有得分和对错，没有解释与指导 | 增加逐题分析、解析、答案对比与改进建议 |
| 错误无法长期纠偏 | 错题不会被系统记住 | 新增错题本并持久化错误记录 |
| 记忆曲线没有作用到测验错误 | 复习系统只覆盖词汇 SRS | 给错题增加独立的 SRS 调度 |
| 薄弱点不可见 | 不知道自己究竟弱在哪类题 | 增加按题型聚合的薄弱点分析 |

#### 结果

测验模块从“结束于提交”演进为“结束后继续驱动学习”的闭环节点，开始真正承担起反馈、纠偏和再训练的职责。

| 平台 | 只有网页 | 移动学习不方便 | 接入 Capacitor + iOS |
| AI | 没有真实可用的大模型接入 | 苏格拉底体验不真实 | 引入本地 AI Bridge + CLI |
| 苏格拉底 | 容易偏离英语学习主线 | 学习目标被稀释 | 增加 guardrails + phase goals |
| 苏格拉底 | 阶段无限扩展，不自然结束 | 体验拖沓，无法闭环 | 增加预算与完成模式 |
| 任务系统 | 苏格拉底完成后未标记任务 | 每日任务状态失真 | 改为状态驱动的统一更新 |
| 测验 | 用户无法顺畅答题 | 学习闭环中断 | 修复组卷和交互鲁棒性 |
| 测后反馈 | 结果页只有分数和对错，缺少逐题分析 | 学生不知道错在哪里，也无法形成长期改进 | 增加逐题展开分析、答案对比、解析和改进建议 |
| 错题修正 | 错题做完即结束，没有长期记忆机制 | 学习很难形成真正纠偏闭环 | 增加错题本、薄弱点分析与基于 SRS 的错题复习 |
| 导航 | 没有完整系统设置入口 | AI 配置不便 | 新增 `SettingsPage` |

### 4.2 已出现并已修复的技术问题

| 问题 | 根因 | 修复 |
|---|---|---|
| `generateWordMatching()` 破坏共享数组 | 原地 `sort()` 变异输入 | 改为 `[...scenes].sort()` |
| 语法题 options 不足 | 错误选项生成过于脆弱 | 增加更多变换 + fallback |
| 组卷时单题报错导致整体失效 | 无异常隔离 | `try/catch` + 题型替补 |
| `startQuiz()` 对空结果无保护 | 缺少防御式编程 | 空检查 + alert 提示 |
| 测验结果无法承接长期纠错 | 测验历史只有聚合分数，没有错题层持久化结构 | 新增错题本存储、SRS 调度和错题复习组卷 |
| 测后反馈过浅 | 结果视图只消费“对/错 + 分数”，没有细粒度分析展示 | 重写结果页，增加逐题分析、薄弱点统计和改进建议 |
| 苏格拉底任务可能漏记 | 完成逻辑分散 | 收口到 `sessionComplete` 统一处理 |
| 路由构建失败 | `App.jsx` 中转义引号错误 | 修正路由 path 字符串 |
| 页面 lint 异常 | 未使用导入/变量、依赖不规范 | 已做多轮清理 |

### 4.3 已出现并记录在案的工程问题

| 问题 | 说明 | 当前状态 |
|---|---|---|
| ESLint 报错较多 | 原型快速迭代期间积累 | 已做一轮系统清理 |
| README 与实现路径曾不完全同步 | 文档保留过 API Key 时代描述，而实现已转向 CLI Bridge | 需要后续继续统一文档 |
| 大量能力集中在前端 | 原型阶段开发快，但后期扩展受限 | 暂时接受，后续需服务端化 |

### 4.4 当前仍待解决 / 应继续记录的问题

这些问题不代表当前不可用，但属于下一阶段必须正视的设计债务。

#### 数据与同步

- 当前用户数据全部在 `localStorage`
- 无登录系统
- 无跨设备同步
- 无服务端画像、推荐和长期统计

#### 音频能力缺失

- 目前仍以文本学习为主
- 缺少原声、TTS、跟读评分、听写模式
- 对“影视英语学习”这个题材来说，这是非常关键的下一阶段能力

#### AI 评分能力不足

- 翻译题和开放题目前没有真正的 LLM 细粒度评分
- `SCENE_CONTEXT` 默认近似给分，准确性有限
- 后续可以把 AI Bridge 继续扩展到“批改器”角色

#### 题型体系仍可继续增强

- `QUIZ_TYPES` 中存在 `SENTENCE_ORDER` 定义，但目前未形成完整生成链路
- 错题本、薄弱点分析、题型推荐尚未建立

#### 内容与版权

- 当前内容属于教学原型样例数据组织方式
- 真正上线前需要明确字幕、音频、视频片段授权范围

## 五、从演进中沉淀出来的设计原则

### 5.1 AI 不是越开放越好，而是越贴近教学目标越好

项目已经验证：

- 让 AI 自由发挥，容易变成“聊天”
- 真正的学习产品需要边界、阶段和目标

所以后续所有 AI 设计都应遵循：

- 明确教学目标
- 控制轮次
- 用状态驱动推进
- 用短反馈 + 单问题降低认知负担

### 5.2 内容结构必须成为统一底座

scene 结构之所以重要，是因为它同时服务：

- 学习页展示
- 测验组题
- 词汇整理
- 苏格拉底追问
- 学习统计

这说明未来任何内容扩展都要优先遵守统一 schema，而不是临时塞字段。

### 5.3 原型阶段可以前端优先，但边界要清楚

当前前端承担了：

- 学习逻辑
- 数据存储
- 组卷评分
- 任务系统
- AI 编排

这在原型阶段是合理的，但后续如果继续扩大，推荐逐步拆分：

- 内容服务
- 用户服务
- 测评服务
- AI 教学服务

### 5.4 真实使用反馈推动的改进价值最高

项目到目前为止最关键的几次进化，都不是“拍脑袋新增功能”，而是来自实际使用暴露的问题：

- 用户没有 API Key → 改用 CLI Bridge
- 苏格拉底偏题 → 增加 guardrails
- 苏格拉底无限扩展 → 加入预算与完成态
- 任务未标记 → 收敛完成状态
- 测验无法答题 → 修复生成与交互鲁棒性

这说明后续迭代应继续坚持：

- 先真实使用
- 再修关键断点
- 再做能力扩展

## 六、当前项目状态总结

### 6.1 当前已经具备的能力

当前项目已经不是单页 demo，而是具备了一套比较完整的原型学习系统：

- 有完整页面结构
- 有第一季内容底座
- 有学习 / 复习 / 测验 / 苏格拉底 / 统计闭环
- 有逐题测后分析、错题本与错题复训闭环
- 有本地 AI Bridge 能力
- 有 iOS 封装基础
- 有本地持久化和成长体系

### 6.2 当前最重要的已知优势

- **主线很清晰**：围绕 Billions 台词学英语，而不是泛泛学英语
- **数据结构统一**：scene 是全链路中心
- **AI 已进入真实可用阶段**：不是假功能，而是能接本地模型做真实追问
- **演进响应很快**：用户提出问题后，系统能直接重构并修复

### 6.3 当前最重要的限制

- 仍然是本地原型架构，不是多端生产架构
- AI 学习能力强于 AI 评分能力
- 文本学习能力强于音频口语能力
- AI Bridge 仍然更像开发期 / 本地运行方案，不是最终移动端分发方案
- 本地存储适合个人试用，不适合产品化规模增长

### 6.4 当前系统 / 架构问题深度分析

这部分聚焦“现在还能跑，但如果继续做大会出问题”的结构性风险。

#### 6.4.1 前端单体承担了过多业务职责

当前前端不仅负责 UI，还直接承担了：

- 学习进度落盘
- 词汇 SRS 调度
- 错题本调度
- 每日任务结算
- 测验评分与历史记录
- Socratic 会话保存
- AI Bridge 配置读取

其中 `src/utils/storage.js` 已经成为事实上的“本地数据库 + 业务编排器”。这在原型期很高效，但问题是：

- 任何新功能都容易继续堆进同一个文件
- 各领域状态边界不清晰
- 交叉副作用变多后，很难判断某个操作会不会影响别的模块
- 后续迁移到 IndexedDB / 云端 / 多端同步时成本会很高

**建议**：把存储层按领域拆分成 `progress`、`quiz`、`srs`、`wrongAnswers`、`dailyPlan`、`aiSettings` 等模块，并引入统一的日期与存储适配器。

#### 6.4.2 页面组件过重，UI 与业务逻辑强耦合

当前几个核心页面都同时承担“界面 + 流程编排 + 状态推进 + 持久化调用”：

- `QuizPage.jsx` 同时处理组卷、答题、提交、错题入库、复习生成、结果分析
- `LearnPage.jsx` 同时处理导航、完成标记、SRS 更新、学习位置保存
- `SocraticPage.jsx` 同时处理会话编排、Bridge 通信、历史恢复、任务标记

这样做的直接问题是：

- 页面越来越长，阅读和修改成本持续上升
- 很难做局部复用
- 很难对业务逻辑做单元测试
- 一个小改动容易带出整页回归问题

**建议**：把页面拆成“展示组件 + 领域 hook/服务”，例如 `useQuizSession()`、`useWrongAnswerReview()`、`useSocraticSession()`。

#### 6.4.3 学习状态缺少统一状态机 / 事件模型

现在不少关键动作是“页面里顺序调用多个函数”完成的。例如测验提交会依次更新：

- quiz history
- daily task
- XP
- wrong answers
- wrong-answer SRS

这类流程目前没有统一事务边界，也没有事件层。只要以后引入云同步、埋点、推荐系统，状态一致性问题就会明显增加。

更关键的是，`src/utils/socratic.js` 里已经有 `evaluateSocraticResponse()` 和 `decideSocraticPhaseAction()` 这样的状态推进辅助函数，但 `SocraticPage.jsx` 运行时仍主要依赖固定轮次推进 phase，说明“设计好的状态机”还没有完整接入“真实运行链路”。

**建议**：逐步把关键学习动作改造成显式 action / reducer / domain-event 流程，让“完成学习”“提交测验”“推进 phase”成为可追踪、可测试的状态转换。

#### 6.4.4 数据与内容层缺少 schema 治理

`scene` 是整个产品的统一底座，这一点是优势；但当前内容仍然主要以静态 JS 常量方式维护，缺少更严格的 schema 验证和内容检查。风险包括：

- 字段完整性依赖人工约束
- typo 或脏数据会直接进入题目生成链路
- 内容越多，越难做批量修复与质量检查

例如当前数据里已经出现过台词文本异常（如 `toeli­minate` 这类内容噪音），这说明内容层需要工程化治理，而不能只靠肉眼巡检。

**建议**：为 `scene / keyword / grammar / episode` 定义正式 schema，并增加内容校验脚本、拼写检查、必填字段检查和题目生成 smoke test。

#### 6.4.5 本地存储方案缺少版本化、迁移与一致性治理

目前系统依赖多个 `localStorage` key，但还没有：

- schema version
- migration 机制
- 数据损坏恢复策略
- 统一的日期工具

另外，日期处理大量依赖 `toISOString().split('T')[0]`，这在跨时区、iOS WebView、本地时间边界附近都可能埋下隐性 bug。`AI_BRIDGE_SETTINGS` 也独立于 `STORAGE_KEYS` 体系，说明存储规范还没有真正收口。

**建议**：建立统一 `storage adapter`，增加版本号、迁移器、日期工具和 key 注册表，为未来迁移到更可靠存储层做准备。

#### 6.4.6 评测能力仍然偏规则驱动，离“智能批改”还有距离

虽然测后分析和错题本已经补上，但评分核心仍然比较粗：

- 翻译题主要靠关键词命中
- `SCENE_CONTEXT` 题仍然是近似给分
- 解析文案多数来自题目生成时的静态字段

这意味着系统现在更擅长“题后反馈展示”，还不够擅长“真正判断学生答案质量”。如果后续希望做开放题、表达迁移、口语或写作训练，这会成为瓶颈。

**建议**：把 AI Bridge 继续扩展为评分器，建立 rubric + 模型评分 + 结果缓存的链路，并区分“生成题目”和“批改答案”两套职责。

#### 6.4.7 工程保障仍然偏薄，没有自动化质量护栏

当前仓库里几乎没有自动化测试，也缺少内容校验、题目生成回归测试和 AI 交互 smoke test。结果是：

- 一些问题只能靠真实点击后才发现
- 数据、逻辑、UI 的回归难以及时暴露
- 随着题型和内容增加，稳定性成本会越来越高

**建议**：优先补三类测试：

- `storage.js` / `quizGenerator.js` / `socratic.js` 的单元测试
- 内容数据的 schema 与生成可用性测试
- 关键学习路径的轻量端到端 smoke test

## 七、建议的下一阶段路线

详细的模块拆分与前后端分离准备方案，见 `docs/architecture-refactor-plan.md`。


### 第一优先级：先做架构减重，再继续堆功能

- 拆分 `storage.js`，按领域收口状态与持久化
- 把 `QuizPage.jsx`、`LearnPage.jsx`、`SocraticPage.jsx` 中的业务流程抽成 hook / service
- 统一日期工具、存储 key 管理、数据版本与迁移机制
- 把“提交测验”“完成学习”“推进 Socratic phase”收敛成可测试的状态动作

### 第二优先级：让错题系统真正驱动自适应学习

- 把错题本、薄弱点分析接入 Dashboard / Stats / Review
- 让测验结果反向推荐 episode、scene 和题型
- 为翻译题 / 开放题接入 AI 评分
- 区分“内容掌握不足”“表达不自然”“语境理解偏差”等错误类型

### 第三优先级：把内容层做成可治理的课程系统

- 为 `scene` 数据建立正式 schema
- 增加内容 lint、拼写检查、字段完整性校验
- 建立内容导入 / 校验 / 发布流水线
- 为题目生成器增加基于内容质量的 smoke test

### 第四优先级：明确移动端与 AI 的长期部署策略

- 评估 AI Bridge 在 iOS 分发场景中的长期可行性
- 明确本地 CLI、远端模型服务、或混合模式的演进路线
- 处理音频资源、离线缓存、同步与恢复
- 为移动端优化输入、复习和错题练习体验

## 八、结论

`Billions English` 的演进过程非常清楚地体现了一条路线：

- **从完整原型出发**
- **被真实使用暴露问题**
- **围绕关键断点不断重构**
- **逐步从“功能存在”走向“体验成立”**

到当前为止，这个项目最有价值的地方不只是“做了哪些功能”，而是已经逐渐沉淀出了一套适用于高语境英语学习产品的设计方法：

- 以 scene 为中心组织内容
- 以学习闭环组织功能
- 以状态机约束 AI
- 以真实问题驱动迭代

后续如果继续推进，建议始终围绕这四条原则扩展，而不是重新回到零散堆功能的路线。
